### 1. `Math.trunc` — `Math.floor`의 사촌 🆕

[문제: 배열 회전시키기](https://school.programmers.co.kr/learn/courses/30/lessons/181850)

> **맨날 소수점 지울 때 `Math.floor` 썼는데, `Math.trunc`도 있다.**

### 차이점 — 음수에서 갈림

```javascript
Math.floor(3.7); //  3
Math.trunc(3.7); //  3      ← 양수일 땐 같음

Math.floor(-3.7); // -4      ← 더 작은 정수로 (내림)
Math.trunc(-3.7); // -3      ← 그냥 소수점 잘라냄
```

| 함수         | 동작                  | 양수 3.7 | 음수 -3.7 |
| ------------ | --------------------- | -------- | --------- |
| `Math.floor` | 더 작은 정수로 (내림) | 3        | **-4**    |
| `Math.trunc` | 소수점 그냥 잘라냄    | 3        | **-3**    |
| `Math.ceil`  | 더 큰 정수로 (올림)   | 4        | -3        |
| `Math.round` | 반올림                | 4        | -4        |

### 코테에서는 어떤 걸 써야 하나?

> **음수 안 나올 때 → `Math.floor` / `Math.trunc` 둘 다 OK** (어차피 결과 같음).
> **음수가 나올 수 있을 때 → 의도에 맞춰 골라야 함.**

```javascript
// 예: "버린다"는 의미 → trunc가 더 직관적
Math.trunc(-7 / 2); // -3 (그냥 잘라냄)
Math.floor(-7 / 2); // -4 (더 작은 쪽으로)
```

> **외울 패턴:** "소수점 잘라낸다" 의도면 `trunc`, "내림"이면 `floor`.
> 코테 1단계 수준에선 거의 양수만 다루니까 둘 다 써도 됨. **무기 하나 추가됨.**

### 2. 영어가 싫어요 — 딕셔너리는 `for...of` 안 됨, `for...in` 써야 함 🆕

[문제: 영어가 싫어요](https://school.programmers.co.kr/learn/courses/30/lessons/81301)

> **첫 번째로 알게된 거.** 처음엔 그냥 풀었는데, 다시 풀다가 신기한 거 발견함.

### 동작 안 한 코드

```javascript
for (var number of dNumbers) {
  console.log(number);
}
// ❌ TypeError: dNumbers is not iterable
```

### 동작한 코드

```javascript
for (var number in dNumbers) {
  console.log(number);
}
// ✅ 키가 출력됨
```

### 핵심 — `for...of`와 `for...in`은 완전 다른 놈

> **`for...of`는 "이터러블"한 애들만 됨** (배열, 문자열, Map, Set 등).
> **`for...in`은 "객체의 키"를 돌려줌** (그래서 일반 객체/딕셔너리에서 씀).

```javascript
const arr = ["a", "b", "c"];
const obj = { zero: 0, one: 1, two: 2 };

// 배열
for (let v of arr) {
  /* 'a', 'b', 'c' */
} // ✅
for (let i in arr) {
  /* '0', '1', '2' */
} // ✅ (근데 권장 X — 인덱스가 문자열로 나옴)

// 객체 (딕셔너리)
for (let v of obj) {
  /* ❌ TypeError */
}
for (let k in obj) {
  /* 'zero', 'one', 'two' */
} // ✅
```

> **외울 패턴:**
>
> - 배열/문자열 돌릴 때 → `of` (값)
> - 객체(딕셔너리) 돌릴 때 → `in` (키)
> - 헷갈리면: **"객체는 키in"** 이렇게 외움

### 그리고 `return +answer` 트릭

```javascript
return +answer; // 문자열 → 숫자 변환
```

> `+`를 문자열 앞에 붙이면 **숫자로 변환**됨. `Number(answer)`랑 똑같은데 더 짧음.
> `parseInt`보다도 짧고, 부호 있는 숫자도 잘 처리함.
> 단점: 처음 보면 "이게 뭐지?" 싶음. 가독성은 `Number()`가 나을 수도.

```javascript
+"123" + // 123
  "12.5" + // 12.5
  "-7" + // -7
  "abc"; // NaN
```

### 3. 콜라 문제 — 주석으로 풀이 전략 적기 = 드디어 통과 🥤

[문제: 콜라 문제](https://school.programmers.co.kr/learn/courses/30/lessons/132267)

> **진짜 싫었던 문제.** 전에 풀 때 멘붕왔던 거 기억남.
> 근데 25일에 배운 **"주석으로 풀이 전략 먼저 적기"** 습관 적용했더니 → **결국 됨.**

### 통과한 코드

```javascript
// a : 콜라 한병으로 바꿔주는 빈병의 개수
// b : a개의 빈병을 가져다 주었을 때 주는 콜라의 개수
// c : 현재 가지고 있는 콜라의 개수
function solution(a, b, n) {
  // 남아있는 빈병의 개수를 정의 할 변수 생성
  let cola = 0;

  // 남아있는 빈병의 개수가 a보다 작을 때 까지 반복문을 돌린다
  while (n >= a) {
    // 빈병을 콜라로 교체한 개수를 가져온다
    cola += Math.floor(n / a) * b;

    // 현재 빈병의 개수를 구해준다
    n = Math.floor(n / a) * b + (n % a);
  }

  return cola;
}
```

### 핵심 — 이번엔 **주석이 진짜 일했음**

> 25일 회고에서 정리한 **"5단계 풀이 루틴"** 그대로 따라감.
>
> 1. 입력/출력 파악 → `a, b, n` 의미 주석으로 박음
> 2. 풀이 전략 주석 → 변수 의미부터 정의, 반복문 종료 조건부터 명시
> 3. 주석 따라 코드 채우기 → 한 줄씩 차근차근

### 콜라 문제의 함정 — `n / a` 한 번 더 쓸 뻔

```javascript
// 빈병을 콜라로 교체한 개수
cola += Math.floor(n / a) * b;

// 새로 가지고 있는 빈병 개수
n = Math.floor(n / a) * b + (n % a);
//  ↑ 새로 받은 콜라(=빈병) ↑ 안 바꾼 자투리 빈병
```

> **두 줄에서 `Math.floor(n / a)`를 두 번 쓰는 게 살짝 비효율.**
> 하지만 **가독성이 더 중요**하다고 판단. 변수에 한 번 담아도 됐는데, 일단 통과한 코드로 둠.

### 더 깔끔하게 한다면

```javascript
function solution(a, b, n) {
  let cola = 0;
  while (n >= a) {
    const exchanged = Math.floor(n / a); // 교환한 횟수
    cola += exchanged * b;
    n = exchanged * b + (n % a);
  }
  return cola;
}
```

> **변수 이름을 박아두면 의도가 명확해짐.** 다음엔 이렇게.

### 회고 — 왜 이번엔 풀렸나?

> **답은 "준비가 됐기 때문".**
> 전에 안 풀린 이유는 머릿속에서 흐름을 짜려고 해서 → 변수 의미가 꼬이고 → 무한루프.
> 이번엔 **주석으로 변수 의미부터 박아놓고 시작** → 헷갈릴 일이 없었음.
> **25일에 배운 습관이 4일 만에 본전 뽑음.** 좋은 습관은 빠르게 효과 옴.

### 근데 찜찜한 거

> **"전에 풀었던 거라서 그런가 뭔가 찜찜함."** ← 이거 진짜 중요한 신호.
> 손이 푼 게 아니라 **기억이 푼 거**라면 다시 풀어야 함. 다음에 이 패턴(분수 floor + 누적) 처음 보는 문제로 만나봐야 진짜 자기 거임.

### 4. K번째 수 — 무난하게 통과 ✅

[문제: K번째 수](https://school.programmers.co.kr/learn/courses/30/lessons/42748)

> 두 번째 풀이라 무난함. **`slice` + `sort` + `[k-1]`** 패턴.

```javascript
function solution(array, commands) {
  return commands.map(
    ([i, j, k]) => array.slice(i - 1, j).sort((a, b) => a - b)[k - 1],
  );
}
```

### 복습 포인트

- `slice(i-1, j)` → 1-based 인덱스를 0-based로 (코테에서 자주 나옴)
- `sort((a,b) => a-b)` → **숫자 오름차순 정렬** (인자 없으면 사전순이라 함정)
- 구조 분해 `([i, j, k])` → 배열 안의 배열 깔끔하게 받기

> **외울 만한 패턴 하나 늘었음.** "잘라서 → 정렬해서 → 인덱스 뽑기".

### 5. 푸드 파이트 대회 — 무난하게 통과 ✅

[문제: 푸드 파이트 대회](https://school.programmers.co.kr/learn/courses/30/lessons/134240)

> 이것도 두 번째라 무난. **반 만들고 → 거꾸로 붙이고 → 가운데 0 박기** 패턴.

### 복습 포인트

- 문자열 만들기 → `repeat()` 메서드
- 거꾸로 → `split('').reverse().join('')`
- 가운데 합치기 → 그냥 `+`로 이어붙이기

```javascript
function solution(food) {
  let left = "";
  for (let i = 1; i < food.length; i++) {
    left += String(i).repeat(Math.floor(food[i] / 2));
  }
  return left + "0" + left.split("").reverse().join("");
}
```

> **이 문제 핵심 깨달음:** 대칭 구조 문제는 **"반만 만들고 뒤집어서 붙이기"**가 정석.
> 처음부터 양쪽 만들려고 하면 인덱스 꼬임.

## 🤔 오늘의 진짜 고민 1 — 복습은 어떻게 해야 하나?

> **8일차에 본인이 직접 던진 질문.** 짚고 가야 함.

### 본인의 고민

- 전에 풀었던 거 다시 푸는 게 맞는 건지
- 일주일치를 하루에 복습? 아니면 다른 방식?
- 시간 너무 길어질 거 같아서 회고만 눈으로 쓱 보고 넘어감
- **"매일 회고 다시 읽기"가 모든 회고를 매일 보라는 건지?**

### 결론 — 복습 자체는 **맞음**, **방식과 범위**가 핵심

> **"전에 풀었던 거 다시 푸는 게 맞아?"** → **맞음.** 코테는 패턴 게임이라서 같은 패턴 여러 번 만나야 자기 거가 됨.
> 근데 **"똑같이 다시 풀기"는 비효율**. 손이 기억하는 거지 뇌가 기억하는 게 아니기 때문.

### "회고 매일 다시 읽기" — 모든 회고를 매일 본다는 게 아님 ⚠️

> **이거 명확히 짚자.** 8일치 회고를 매일 다 읽으라는 게 아님. 그러면 양 누적돼서 결국 안 함.

### 회고 복습 — 3단계 시간 범위

| 범위                       | 빈도          | 시간           |
| -------------------------- | ------------- | -------------- |
| **최근 2~3일치 회고**      | 매일          | 5분 (스크롤만) |
| **이번 주 회고 통합 훑기** | 주 1회 (주말) | 15~20분        |
| **2주 이상 된 회고**       | 필요할 때만   | 검색해서 찾기  |

> **핵심: 매일 보는 건 "최근 거"만.** 옛날 거는 잊을만하면 보면 됨.
> 8일차인 지금은 "어제 + 그제 회고만 매일 5분" 정도면 충분.

### 옛날 회고는 어떻게?

> **검색용 인덱스로 활용.**
> 코딩 중에 "어 이거 본 적 있는데..." 싶으면 회고 폴더에서 키워드 검색.
> 예: "정렬 어떻게 했지?" → 폴더에서 `sort` 검색 → 25일 회고 나옴 → 거기서 복습.
> **"매일 다시 읽기"가 아니라 "필요할 때 꺼내 쓰기".**

### 복습 활동 — 4가지 우선순위

| 활동                  | 시간 비율 | 빈도            |
| --------------------- | --------- | --------------- |
| 새 문제 풀기 (메인)   | **70%**   | 매일            |
| 최근 회고 훑기        | **15%**   | 매일 5분        |
| 옛날 회고 검색        | **5%**    | 코딩 중 필요 시 |
| 헤맸던 문제 다시 풀기 | **10%**   | 주 1~2회        |

### 본인이 오늘 한 건 어땠나?

> **2일 쉬고 복귀한 첫날 = 복습 100%** 였는데, **이건 예외 상황이라 OK.**
> 컨디션 회복용으로 좋았음. **근데 내일부터는 비율 다시 맞춰야 함.**
> "회고만 쓱 봄" → 이거 정확히 맞는 방식임. 시간 줄이고 핵심만 짚는 거.

### 오늘 깨달은 거 — "찜찜함"의 의미

> 콜라 풀고 **"전에 풀어서 그런가 찜찜함"** 라고 적었는데, **이게 신호임.**
> 찜찜함 = "내가 푼 게 아니라 기억이 푼 것 같다" = **다음에 비슷한 새 문제로 검증 필요.**
> 이걸 알아챈 것만으로도 **메타인지 수준이 올라간 거.** 좋은 신호 ✨

## 🤔 오늘의 진짜 고민 2 — 알고리즘/자료구조는 언제 시작?

> **이것도 본인이 던진 질문.** 짚고 가자.

### 결론 먼저 — **지금은 아직 일러요. 2~3주 더 1단계.**

> 8일차에 자료구조 시작하면 **적용할 데가 없어서 까먹음**. 1단계는 자료구조 거의 안 씀.
> **2단계부터 스택/큐/해시/완전탐색이 강제로 필요해짐** → 그때 배우는 게 흡수율 최고.

### 권장 로드맵

```
[현재] 8일차, 1단계 진행 중
   ↓
[2~3주차] 1단계 30~50문제 푼 후
   ↓
[3~4주차] 2단계 진입 + 1단계 헤맸던 거 다시 풀기
   ↓ ← 여기서 자료구조 시작
[4주차~] 자료구조 (스택/큐 → 해시 → 트리 → 그래프 순)
   ↓
[6주차~] 알고리즘 (정렬/탐색 → DFS/BFS → DP)
```

### 왜 2단계 진입 시점인가?

> **자료구조는 "배우고 쓰는" 게 아니라 "필요해서 배우는" 게 효율적.**
> 2단계 가면 자연스럽게:
>
> - "괄호 짝 맞추기" → 스택 필요
> - "프로세스 처리 순서" → 큐 필요
> - "전화번호 목록" → 해시 필요
>
> **문제가 자료구조를 강제할 때 배우는 게 가장 박힘.** 미리 배우면 추상적이라 안 와닿음.

### 시작 순서 — 자료구조 먼저, 알고리즘 나중

> 이유: **알고리즘이 자료구조를 사용함.** DFS는 스택/큐 알아야 이해됨. DP는 배열을 잘 다뤄야 함.

### 자료구조 학습 순서 (입문)

1. **스택 (Stack)** — LIFO. 괄호 맞추기, 뒤로가기 같은 데 씀
2. **큐 (Queue)** — FIFO. 순서대로 처리할 때 씀
3. **해시 (Hash/Map)** — 빠른 검색. 자바스크립트의 `Map` / `Object`
4. **트리** — 계층 구조. 폴더, 댓글
5. **그래프** — 관계. 친구 관계, 길찾기

### 알고리즘 학습 순서 (입문)

1. **정렬** — 이미 어느정도 함 (`sort`)
2. **이진 탐색 (Binary Search)** — 정렬된 배열에서 빠르게 찾기
3. **완전 탐색 / 브루트포스** — 전부 다 해보기 (1단계에서도 자주 나옴)
4. **DFS / BFS** — 탐색 알고리즘. 그래프/트리에서 씀
5. **DP (동적 계획법)** — 큰 문제를 작은 문제로 쪼개기. 어려움.

### 학습 자료 추천 (시작할 때 참고)

> **지금 정해두지 말고, 2단계 들어갔을 때 다시 결정.** 그 시점에 본인 스타일에 맞는 거 고르기.

대표 옵션:

- **유튜브** — 동빈나, 엔지니어대한민국 (한국어, 무료)
- **책** — "코딩 테스트를 위한 자료 구조와 알고리즘 with 자바스크립트" 류
- **인프런** — 강의 형식 선호하면 (유료지만 체계적)
- **프로그래머스 강의** — 풀던 사이트라 연동 자연스러움

> **지금은 결정 미뤄도 됨.** 1단계 더 풀면서 본인이 어떤 스타일로 배우는지 파악하기.

### 그럼 지금은 뭐 해?

> **딴 거 신경 쓰지 말고 1단계 새 문제 계속 풀기.**
> 1단계만 잘 풀어도 어떤 회사 코테는 통과 가능. 자료구조 미리 안 배워도 됨.
> **"1단계 어느 정도 무난해졌다"는 감 = 30~50문제 풀고 헤매는 빈도 줄었을 때.**

## 8일차 정리 — 학습 방향성

### ✅ 오늘 한 거

- 머리 식히기로 0단계 → `Math.trunc` 발견
- 영어가 싫어요 → `for...in` vs `for...of` 차이 발견
- 콜라 → 주석 전략으로 결국 통과
- K번째 수, 푸드파이트 → 무난 통과
- **"복습 어떻게 할까"** 본인이 메타인지 질문 던짐
- **"자료구조/알고리즘 언제 시작?"** 로드맵 정리됨

### 🆕 새로 배운 거

- `Math.trunc` — 소수점 그냥 잘라냄 (음수에서 `floor`와 다름)
- `for...of`는 이터러블만, **객체는 `for...in`**
- `+answer` → 문자열 → 숫자 변환 트릭

### 🔁 다시 박은 패턴

- `Math.floor(n / a)` 누적 (콜라)
- `slice → sort → [k-1]` (K번째)
- `repeat → reverse → join` (대칭 구조)

### 🎯 내일부터 복습 루틴 (확정)

1. **메인은 새 문제** (70%) — 1단계 계속
2. **최근 2~3일 회고만 매일 5분 훑기** (15%) — 절대 전체 다 보지 말 것
3. **헤맸던 문제만 주 1~2회 다시 풀기** (10%) — 콜라 같은 거
4. **옛날 회고는 검색용**으로만 (5%) — 필요할 때 꺼내 쓰기

### 🗺️ 자료구조/알고리즘 — 지금은 보류

- 현재 8일차 → 너무 이름
- **2~3주 더 1단계 풀고**, 2단계 진입 시점에 시작
- 순서: 스택/큐 → 해시 → 트리 → 그래프 → 정렬/탐색 → DFS/BFS → DP

### 💬 한마디

> 비염 + 코감기로 2일 쉰 거, 처음엔 짜증났는데 **돌아와서 복습하니까 오히려 정리됨.**
> 콜라 문제 풀린 거 — 25일 습관(주석 전략)이 4일 만에 효과 본 거. **좋은 습관은 빠르게 본전 뽑음.**
> "찜찜함"을 인지한 것 = 메타인지 시동 걸림. 이게 8일차의 진짜 수확.
> **복습 = 옛날 문제 다시 푸는 게 아니라, 옛날 패턴을 새 문제에 써보는 것.**
> **자료구조/알고리즘 = 미리 배우는 게 아니라, 문제가 강제할 때 배우는 것.**
> 두 가지 큰 방향성 잡힌 거. 오늘도 남는 장사 🚀

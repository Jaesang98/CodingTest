### 1. 🆕 배열 회전 — 스왑으로 풀려다가 문제 잘못 봄 사건 😅

[문제: n번째 원소까지](https://school.programmers.co.kr/learn/courses/30/lessons/181891?language=javascript)

### 처음 시도한 코드 (스왑으로 풀려고 함)

```javascript
function solution(num_list, n) {
  [num_list[n], num_list[n + 1]] = [num_list[n + 1], num_list[n]];
  return num_list;
}
```

> 본인 멘트: **"이거 왜 안 되지? 전에 이런 거는 잘 되었는데 똑같이 했는데 안 됨"**

### 참고 — 전에 잘 됐던 스왑 코드 (다른 문제)

```javascript
function solution(my_string, num1, num2) {
  my_string = my_string.split("");
  [my_string[num1], my_string[num2]] = [my_string[num2], my_string[num1]];
  return my_string.join("");
}
```

### 🔥 본인이 직접 발견한 핵심

> 본인 멘트: **"아 아니네 스왑은 잘 되는데 내가 문제를 잘못 봄 ㅋㅋ"**
>
> **문제 잘못 본 거 빨리 캐치한 게 핵심.** 코드가 안 되면 보통 코드 의심하는데,
> **문제 자체를 의심한 것** = 메타인지 작동 중.

### 최종 풀이 — `slice` + spread

```javascript
function solution(num_list, n) {
  return [...num_list.slice(n), ...num_list.slice(0, n)];
}
```

> **외울 패턴:**
>
> - **배열 회전**은 `slice` 두 번 + spread로 합치기 → 원본 안 건드림 ✨
> - 어제(10일차) `slice(n)`이 "n부터 끝까지" 줍는 거 박은 게 오늘 자동 발동
> - 코드 안 되면 **문제부터 다시 읽기** — 회고에 박을 만한 습관

### ⚠️ 비구조화 할당 스왑 + 세미콜론 함정

```javascript
// 이게 안 되는 진짜 이유는 사실 "문제를 잘못 봤다"였지만
// 비구조화 스왑 자체에는 함정이 있음
function solution(num_list, n) {
  [num_list[n], num_list[n + 1]] = [num_list[n + 1], num_list[n]];
  //            ↑
  // 위 줄이 세미콜론 없이 끝나면 [ 가 이전 식의 인덱싱으로 해석될 수 있음
  return num_list;
}

// 그래서 본인이 다른 문제에서 ;[...] 이렇게 앞에 세미콜론 박은 거였음
[my_string[num1], my_string[num2]] = [my_string[num2], my_string[num1]];
```

> **외울 패턴:**
>
> - 줄 시작이 `[` 또는 `(` 이면 **앞에 세미콜론** 박기 (ASI 함정)
> - 다만 이 문제에서는 함수 첫 줄이라 세미콜론 이슈는 없었음 → 진짜 원인은 **문제 오독**

### 2. 🆕 문자열 내림차순 — 대소문자 분리 정렬

[문제: 문자열 내림차순으로 배치하기](https://school.programmers.co.kr/learn/courses/30/lessons/12917)

```javascript
function solution(s) {
  var result1 = s
    .split("")
    .filter((e) => e.toLowerCase() === e)
    .sort((a, b) => b.localeCompare(a))
    .join("");
  var result2 = s
    .split("")
    .filter((e) => e.toUpperCase() === e)
    .sort((a, b) => b.localeCompare(a))
    .join("");

  return result1 + result2;
}
```

> 본인 멘트: **"문자열 sort, 알파벳 대소문자 구분 이제 좀 잘 익힌 듯"**

### 핵심 트릭 분석

```javascript
// 1. 대소문자 판별 — toLowerCase()/toUpperCase()와 비교
"a".toLowerCase() === "a"; // true → 소문자
"A".toLowerCase() ===
  "A"[ // false → 대문자
    // 2. 문자열 내림차순 — localeCompare 뒤집기
    ("c", "a", "b")
  ]
    .sort((a, b) => b.localeCompare(a)); // ['c','b','a']

// 3. 문제의 트릭: 소문자 먼저, 대문자 나중에 (아스키 순)
//    → 소문자 내림차순 결과 + 대문자 내림차순 결과를 그냥 이어붙이면 됨
```

> **외울 패턴:**
>
> - **대소문자 판별**: `e === e.toLowerCase()` (소문자), `e === e.toUpperCase()` (대문자)
> - **문자열 정렬**: `localeCompare` (오름차순), `b.localeCompare(a)` (내림차순)
> - 숫자 정렬과 헷갈리지 말 것 — 문자열은 `localeCompare`가 안전

### ⚠️ 사실 이 코드도 더 줄일 수 있음 — 트레이드오프 체크

```javascript
// 한 번 split해서 대/소문자 한 번에 분리하면 split 1번만 하면 됨
function solution(s) {
  const arr = s.split("").sort((a, b) => b.localeCompare(a));
  const lower = arr.filter((e) => e === e.toLowerCase()).join("");
  const upper = arr.filter((e) => e === e.toUpperCase()).join("");
  return lower + upper;
}

// 또는 더 짧게 — 정렬 한 번으로 끝
function solution(s) {
  return s
    .split("")
    .sort((a, b) => b.localeCompare(a))
    .join("");
  // ← 사실 이 문제는 그냥 전체 내림차순이면 끝남
  //    소문자 > 대문자 (아스키 코드상) 이라 자연스럽게 분리됨
}
```

> 🤔 **회고 포인트:** 본인 코드도 동작 OK, 의도도 명확함.
> 다만 **"split 두 번 = 같은 작업 두 번"** 이라는 어제 룰 적용해보면 줄일 여지 있음.
> 어제 "반복문 안 변하지 않는 값 빼라" → 오늘 "같은 split 두 번 하지 마라" 로 확장 가능.

### 3. 🆕 `0x16` 함정 — `+` 와 `Number()`가 16진수도 먹는다 ㅅㅂ

[문제: 문자열 다루기 기본](https://school.programmers.co.kr/learn/courses/30/lessons/12918)

### 처음 풀이 (테스트 11번에서 깨짐)

```javascript
// 처음엔 한 줄짜리로 풀었던 듯
// "길이 4 or 6 이고 전부 숫자인가?" → +s 로 변환되면 숫자
```

### 함정 — `0x16`도 `+` 붙이면 숫자가 됨

```javascript
+"0x16"; // 22 ❌ (숫자처럼 보임, 근데 우리가 원하는 게 아님)
Number("0x16") + // 22 ❌
  "123" + // 123 ✅
  "abc" + // NaN
  "0"; // 0 ← 이것도 함정! falsy

// 즉, "+e" 만 검사하면 "0x16"도 통과해버림
```

### 수정한 코드

```javascript
function solution(s) {
  var answer = true;
  if (!(s.length === 4 || s.length === 6)) {
    answer = false;
  }

  s.split("").forEach((e) => {
    if (!(+e || +e === 0)) {
      answer = false;
    }
  });

  return answer;
}
```

### 🔥 핵심 트릭 — `+e || +e === 0`

```javascript
// 왜 이렇게 썼는가?
+"0" + // 0 → falsy! 그래서 +e만 쓰면 "0"이 통과 못함
  "0" ===
  0; // true → 0인 경우는 따로 살림

// 그래서 (+e || +e === 0) = "숫자로 변환되는데, 0도 OK"
// 한 글자씩 검사하니까 "0x16"이 들어와도 'x'에서 NaN 걸려서 false
```

### 다른 사람들의 더 깔끔한 방법 — 정규식 / 0~9 직접 검사

```javascript
// 방법 1: 정규식
function solution(s) {
  return (s.length === 4 || s.length === 6) && /^[0-9]+$/.test(s);
}

// 방법 2: 0~9 사이인지 직접 검사
function solution(s) {
  if (s.length !== 4 && s.length !== 6) return false;
  return s.split("").every((e) => e >= "0" && e <= "9");
}

// 방법 3: isNaN 활용 (이건 0x16 함정 똑같이 있음 ⚠️)
```

> **외울 패턴:**
>
> - **`+`나 `Number()`는 16진수(`0x16`), 8진수(`0o17`), 2진수(`0b10`) 다 먹음**
> - **순수 숫자 문자열인지** 검사하려면 → 정규식 `/^[0-9]+$/` 또는 문자별 비교
> - 한 글자씩 검사하면 `0x16` 같은 거 자연스럽게 걸러짐 (본인 코드처럼)
> - **`+e || +e === 0`** 패턴 외워두기 — "숫자로 변환되는가, 0도 포함" 의미

> 🤔 **회고 포인트:** GPT 안 쓰고 본인이 끝까지 디버깅한 점이 좋음.
> 이런 함정 한 번 직접 부딪혀야 평생 안 잊음. 정규식 쓰면 한 줄이지만,
> **"`+`로 변환되는 별의별 케이스"** 라는 자바스크립트의 변태성을 체득한 게 진짜 자산.

### 4. 🆕 `filter`에서 `return e.toString()`이 왜 통과되는가 — Truthy/Falsy 사건 🔍

[문제: 같은 숫자는 싫어 (스택/큐)](https://school.programmers.co.kr/learn/courses/30/lessons/12906)

### 본인 코드 — `e.toString()` 리턴

```javascript
function solution(arr) {
  return arr.filter((e, i) => {
    if (!(arr[i] === arr[i + 1])) {
      return e.toString();
    }
  });
}
```

> 본인 멘트: **"이게 왜 되지??? return e만 썼을 때는 0을 없애고 return해서 toString으로 변환한 뒤 다시 숫자로 바꿔야지 했는데 걍 통과가 되네;;;"**

### 다른 사람 풀이 — 비교식 그대로 리턴

```javascript
function solution(arr) {
  return arr.filter((val, index) => val != arr[index + 1]);
}
```

### 본인 최종 풀이 — forEach + push (가장 명시적)

```javascript
function solution(arr) {
  var answer = [];
  arr.forEach((e, i) => {
    if (!(arr[i] === arr[i + 1])) {
      answer.push(arr[i]);
    }
  });
  return answer;
}
```

### 🔥 핵심 — `filter`는 콜백 리턴값의 **truthy/falsy**만 본다

```javascript
// filter는 리턴값을 Boolean으로 변환해서 판단
// truthy → 포함, falsy → 제외

// JS의 falsy 값들 (이것만 빼면 다 truthy):
false, 0, "", null, undefined, NaN, 0n

// 그래서 본인 코드 분석:
[1, 1, 3, 0, 4, 4, 3].filter((e, i) => {
    if(/* 다음과 다르면 */) {
        return e.toString()
        // 1 → "1" → truthy ✅
        // 3 → "3" → truthy ✅
        // 0 → "0" → truthy ✅ (문자열 "0"은 truthy!)
        // 4 → "4" → truthy ✅
    }
    // 같으면 if 안 들어감 → 함수가 undefined 리턴 → falsy → 제외 ✅
})
```

### ⚠️ 만약 `return e` 였다면 — 본인이 우려한 시나리오

```javascript
[1, 1, 3, 0, 4].filter((e, i) => {
    if(/* 다음과 다르면 */) {
        return e
        // 1 → truthy ✅
        // 0 → falsy ❌  ← 0을 잃어버림!
    }
})
```

> 본인이 **정확히 이 함정을 미리 인지하고** `toString()` 으로 우회한 거임.
> "0이 falsy라 빠질 거 같다" → toString하면 `"0"` (truthy) → 살아남음.
> **숨은 의도가 있는 코드.** 본인이 풀 때 이미 알고 있었음. 굿.

### 왜 다른 사람 풀이 `val != arr[index+1]` 도 되는가

```javascript
// 비교 연산자는 항상 boolean 리턴
val != arr[index + 1];
// → true (다름) → 포함
// → false (같음) → 제외
// 깔끔함 ✨
```

### 🆕 추가 깨달음 — "그러면 `toString()` 리턴하면 결과 배열도 문자열 아냐?" 🤔

> 본인 의문: **"`return e.toString()` 하면 배열에 문자열이 와야 하는 거 아니야?"**
>
> **이게 진짜 좋은 질문이었음.** 여기서 `filter`의 본질이 갈림.

### `filter` vs `map` — 콜백 리턴값의 역할이 완전 다름

```javascript
// filter — 콜백 리턴값 = "포함할지 말지" (truthy/falsy만 봄)
[1, 0, 3]
  .filter((e) => e.toString())
  [
    // → [1, 0, 3]  ← 원본 그대로! "1","0","3" 아님!!

    // map — 콜백 리턴값 = "변환된 새 값"
    (1, 0, 3)
  ].map((e) => e.toString());
// → ["1", "0", "3"]  ← 이건 진짜 문자열로 바뀜
```

### `filter` 내부 동작 (의사 코드)

```javascript
function filter(arr, callback) {
  const result = [];
  for (let i = 0; i < arr.length; i++) {
    if (callback(arr[i], i)) {
      // ← 리턴값은 boolean으로만 씀
      result.push(arr[i]); // ← 넣는 건 원본 arr[i]!
    }
  }
  return result;
}
```

> **핵심**: `result.push(arr[i])` — **콜백 리턴값을 넣는 게 아니라 원본을 넣음.**
>
> 그래서 본인 코드에서:
>
> - `e.toString()` → `"0"` → truthy → "통과 OK" 신호
> - 결과 배열에는 **원본 숫자 `0`이 push됨**
> - 그래서 결과는 `[1, 3, 0, 4, 3]` (다 숫자 ✅)

### 🔑 외우는 비유

```javascript
// filter = 체 (sieve) 🪤
// "이거 통과시킬까 말까?" 만 결정 → 통과한 건 원본 그대로

// map = 공장 (factory) 🏭
// "이거 어떤 모양으로 바꿀까?" → 변환된 값이 결과
```

### 콜백 리턴 타입으로도 구분 가능

```javascript
// filter는 보통 비교식이 자연스러움 (boolean 리턴)
arr.filter((e) => e > 0);
arr.filter((e) => e !== null);
arr.filter((e) => e.length > 3);

// map은 보통 변환식이 자연스러움 (새 값 리턴)
arr.map((e) => e * 2);
arr.map((e) => e.toUpperCase());
arr.map((e) => ({ ...e, active: true }));
```

> **외울 패턴:**
>
> - **"길이 변하면 filter, 모양 변하면 map"** 🎯
>   - filter: 결과 배열 길이 ≤ 원본 길이 (걸러내니까)
>   - map: 결과 배열 길이 = 원본 길이 (1:1 변환)
> - `filter` 콜백 → **boolean 리턴**이 자연스러움
> - `map` 콜백 → **새 값 리턴**이 자연스러움
> - filter 콜백에 `return 새값` 쓰고 있으면 **냄새 의심** → "거르는 거야, 변환하는 거야?" 자문

### 🔥 본인이 무의식적으로 맞았던 이유

> 본인 멘트: **"map이랑 헷갈림 ㅋㅋ"**
>
> 헷갈렸다는 것 자체가 **언어 감각이 작동하는 증거**임.
> filter인데 콜백이 map처럼 새 값 리턴하니까
> "어 이거 문자열 배열 나와야 되는 거 아냐?" 라고 본능적으로 느낀 거.
> **그 본능이 맞는 방향임** — 다음엔 그 감각 따라서 바로 `boolean` 리턴으로 바꾸면 됨.

### 그래서 가장 깔끔한 버전은

```javascript
// filter답게 — boolean 리턴
function solution(arr) {
  return arr.filter((e, i) => arr[i] !== arr[i + 1]);
}
// "거르는 거구나" 한 눈에 읽힘 ✨
```

| -------------------------------- | ---- | ------------ | -------------------------- |
| 본인 1차 (`return e.toString()`) | OK   | 🤔 의도 숨김 | 0이 falsy인 걸 알고 우회   |
| 다른 사람 (`val != arr[i+1]`)    | OK   | ✨ 명확      | boolean 반환이라 함정 없음 |
| 본인 최종 (forEach + push)       | OK   | ✨ 가장 명확 | 함정 자체가 없음           |

> **외울 패턴:**
>
> - **`filter` 콜백은 truthy/falsy로 판단** — `0`, `""`, `null`, `undefined`, `NaN` 조심
> - 숫자 배열 다룰 때 `filter`에서 **`return e` 절대 금지** (0 잃어버림)
> - 가장 안전한 건 **비교 연산자로 boolean 리턴** (`val != ...`)
> - 본인 `forEach + push` 풀이도 좋음 — **명시적 = 디버깅 쉬움**

> 🤔 **회고 포인트:** "이게 왜 되지?" 라고 의문 던진 게 핵심.
> 그냥 통과했다고 넘어가지 않고 **원리를 파헤친 것** = 진짜 학습.
> 어제(10일차)에 박은 "원본 변경 메서드 의식하기" 룰이 오늘은
> **"리턴값이 어떤 타입으로 변환되는지 의식하기"** 룰로 확장됨.

### 패턴 — "왜 되지?" 의 가치

>
> 양치기 100문제 푼 사람 중에 **"왜 되지?"** 를 던진 사람만 진짜 성장함.

## 11일차 정리 (낮 편)

### ✅ 오늘 한 거 (낮 분량)

- 0단계 마무리 5문제 (배열 회전 등)
- 1단계 진입 5문제 (문자열 정렬, 0x16 함정, filter 함정 등)
- **GPT 의존도 낮추고 본인 머리로 디버깅** ✨

### 🆕 새로 배운 거

- **배열 회전** = `slice` 두 번 + spread (`[...arr.slice(n), ...arr.slice(0,n)]`)
- **문자열 정렬** = `localeCompare` / 내림차순은 `b.localeCompare(a)`
- **대소문자 판별** = `e === e.toLowerCase()` / `e === e.toUpperCase()`
- **`0x16` 함정** = `+`나 `Number()`는 16진수/8진수/2진수도 먹음
- **`+e || +e === 0`** = "숫자로 변환되는가, 0 포함" 검사 패턴
- **`filter` 콜백은 truthy/falsy 판단** — 0 잃어버리는 함정 주의
- **`filter`는 콜백 리턴값으로 "포함 여부"만 판단, 결과 배열엔 원본 push** ← 오늘 새 깨달음 🔥
- **filter = 체 🪤 / map = 공장 🏭** — "길이 변하면 filter, 모양 변하면 map"
- `filter` 콜백에 `return 새값` 쓰고 있으면 **냄새 의심** → boolean 리턴으로 바꾸기
- 순수 숫자 문자열 검사 → **정규식 `/^[0-9]+$/`** 가 정석

### 🔁 다시 박은 패턴

- 어제(10일차) **"반복문 안 변하지 않는 값 빼라"** → 오늘 "같은 split 두 번 하지 마라"로 확장
- 새벽(11일차) **"다른 사람 풀이 = 참고 자료"** → 낮에 **"왜 이게 되지?"** 의문으로 발전
- **원본 변경 / 짧은 코드 / 트릭** 무조건 좋은 거 아님 (3일째 같은 룰 적용 중)

### ⚠️ 오늘의 실수 / 자각

- 배열 회전 문제 — **문제를 잘못 봐서** 스왑으로 풀려 함 → 빨리 캐치한 건 잘함
- `filter` 에서 `return e.toString()` — 동작은 하지만 **의도 숨긴 코드**
- → 둘 다 **자각 자체는 실수가 아님.** 다음에 안 그러는 게 중요.

### 🎯 새벽 다짐 실천 점검

- ✅ **0단계 10분만** → 5문제로 마무리
- ✅ **1단계 쉬운 거 많이** → 5문제 + 함정 두 개 발견
- ⏳ **1단계 어려운 거 일부** → 다음 세션
- ✅ **"왜 이게 더 짧지? 트레이드오프?"** 던지기 → filter truthy/falsy 추적

### 🎯 다음 다짐

## 0단계 — 머리 회전용 🍱

### 1. 🆕 `eval()` — 문자열 산수식을 그대로 계산해주는 마법

[문제: 문자열 사칙연산 (이항 계산)](https://school.programmers.co.kr/learn/courses/30/lessons/181865?language=javascript)

### 처음 푼 코드 (열심히 분기)

```javascript
function solution(binomial) {
  for (const item of ["+", "-", "*"]) {
    if (binomial.includes(item)) {
      binomial = binomial.split(item);

      if (item === "+") {
        return +binomial[0] + +binomial[1];
      }
      if (item === "-") {
        return +binomial[0] - +binomial[1];
      }
      if (item === "*") {
        return +binomial[0] * +binomial[1];
      }
    }
  }
}
```

> 본인 멘트: **"문자열도 되있는 산수연산?? 그거를 아래처럼 해도되네 ㄷㄷ"**

### 다른 사람 풀이 — `eval` 한 줄

```javascript
function solution(binomial) {
  return eval(binomial);
}
```

### `eval` 기본 사용법

```javascript
eval("1 + 2"); // 3
eval("3 * 4"); // 12
eval("10 - 5 + 2"); // 7
eval("'hello' + ' world'"); // 'hello world'

// 변수도 됨 (스코프 안의 변수 접근 가능)
const x = 10;
eval("x + 5"); // 15
```

### ⚠️ `eval`의 함정 — 마냥 좋은 게 아님

```javascript
// 1. 보안 위험 — 사용자 입력을 그대로 eval 하면 임의 코드 실행
const userInput = "alert('hacked')";
eval(userInput); // ❌ 위험!

// 2. 성능 — 자바스크립트 엔진 최적화 방해
// 3. 디버깅 어려움 — 에러 추적이 힘듦
```

> **외울 패턴:**
>
> - 코테에서는 **간단한 산수식 계산**할 때 빠른 트릭으로 OK
> - **실무에서는 절대 쓰지 말 것.** MDN도 "Never use eval()" 라고 박아놓음
> - 안전한 대안: `Function` 생성자, 또는 직접 파싱
> - 본인 코드처럼 **명시적 분기**도 충분히 좋음. 의도가 명확하니까.

> 🤔 **회고 포인트:** `eval` 한 줄이 멋있어 보이지만, 본인 처음 푼 코드도 **나쁘지 않음**.
> 코테 한정으로는 `eval` 알아두되, **"항상 짧은 게 좋은 건 아니다"** 라는 감각도 같이 챙기기.

### 2. 🆕 `reduce` 한 번으로 충분한데 두 번 쓴 자만 사건 😤

[문제: 길이에 따라 다른 연산](https://school.programmers.co.kr/learn/courses/30/lessons/181879?language=javascript)

### 처음 푼 코드 (reduce 두 번)

```javascript
function solution(num_list) {
  return num_list.length > 10
    ? num_list.reduce((a, b) => (a += b), 0)
    : num_list.reduce((a, b) => (a *= b), 1);
}
```

> 본인 멘트: **"어떤 사람 풀이를 보니 reduce를 한번만 사용해도되네 ㅅㅂ"**

### 다른 사람 풀이 — `reduce` 한 번

```javascript
function solution(num_list) {
  return num_list.reduce((a, b) =>
    num_list.length > 10 ? (a += b) : (a *= b),
  );
}
```

### 🔥 본인이 직접 발견한 핵심

> 본인 멘트: **"아 새벽이라 그런가 아니다 좀 자만했다"**
>
> **이 한 줄이 오늘의 진짜 수확.** 코드 짧아진 거보다 **자기 인식**이 훨씬 큰 자산.

### 차이 분석 — 왜 한 번이면 충분한가

```javascript
// 본인 코드 — 분기가 reduce 바깥
length > 10 ? reduce(+) : reduce(*)
// → 분기 한 번, reduce 두 번 정의 (실제 호출은 1번)

// 개선 코드 — 분기가 reduce 안쪽
reduce((a,b) => length > 10 ? a+b : a*b)
// → 분기는 매 반복마다 일어나지만, reduce 정의는 한 번
```

### ⚠️ 잠깐 — 이거 진짜 더 좋은 코드인가?

> **솔직하게 보면 토론거리 있음.**

```javascript
// 본인 코드의 장점:
// - length 비교가 1번만 일어남 (reduce 시작 전)
// - 의도가 명확: "길면 합, 짧으면 곱"

// 개선 코드의 장점:
// - 줄 수 짧음
// - reduce 한 번

// 개선 코드의 단점:
// - num_list.length > 10 비교가 매 반복마다 일어남 (n번)
// - 어제 배운 "반복문 안에서 변하지 않는 값은 빼라" 룰 위반!!
```

### 🚨 어제 배운 룰 적용해보면 — 본인 원래 코드가 더 나을 수도?

> **어제(10일차) 회고에서 박은 룰:**
> _"반복문 안에서 변하지 않는 값은 변수로 빼라"_
>
> `num_list.length > 10` 은 reduce 도는 동안 **절대 안 변함.**
> 그러면 매 반복마다 비교하는 건 낭비임.

```javascript
// 가장 좋은 버전 — 어제 배운 거 적용
function solution(num_list) {
  const isLong = num_list.length > 10;
  return num_list.reduce((a, b) => (isLong ? a + b : a * b), isLong ? 0 : 1);
}

// 또는 본인 원래 코드 — 어차피 분기 1번이라 이것도 좋음
function solution(num_list) {
  return num_list.length > 10
    ? num_list.reduce((a, b) => a + b, 0)
    : num_list.reduce((a, b) => a * b, 1);
}
```

> **외울 패턴:**
>
> - "다른 사람 풀이가 짧다 = 무조건 좋다" ❌
> - **짧은 코드 = 가독성 ✨ vs 매 반복 분기 = 성능 ❌** 트레이드오프 있음
> - 코테 통과 기준에서는 둘 다 OK. **하지만 사고는 항상 깊게.**
> - 어제 배운 룰을 새 문제에 자동 적용하는 게 **진짜 학습.**

### ⚠️ 추가 — `reduce` 콜백 안의 `a += b` 표기법

```javascript
// 셋 다 같은 결과
arr.reduce((a, b) => (a += b), 0);
arr.reduce((a, b) => a + b, 0); // ← 이게 더 깔끔
arr.reduce((a, b) => (a = a + b), 0);
```

> **외울 패턴:**
>
> - `reduce` 콜백 안에서는 `a += b` 보다 `a + b` 가 의도 명확
> - `+=` 는 `a`를 변경한다는 느낌인데, 어차피 다음 반복에서 `a`는 새 값으로 들어옴
> - 화살표 함수 + 명시적 return 없으면 → **연산식 하나가 깔끔**

### 패턴 — "다른 사람 풀이"를 대하는 태도 업데이트

이전:

- 다른 사람 풀이 = 짧으면 정답, 베끼기

지금:

- 다른 사람 풀이 = **참고 자료**
- 짧다고 무조건 좋은 거 아님 → **왜 짧은지, 어떤 트레이드오프가 있는지** 분석
- 본인 코드도 의도 명확하면 그 자체로 가치 있음

> 🎯 **`eval` 사건이 좋은 예시.** 한 줄짜리 `eval`이 멋있지만, 본인 분기 코드는 **읽기 쉽고 안전함**.
> 코테에서는 트릭, 실무에서는 본인 스타일이 정답일 수도.

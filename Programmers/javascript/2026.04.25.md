## 0단계 — 머리 식히기

### 1. `reduce`에서 `Number(a) += ...` 왜 안 되냐? 🔥

[문제: 자릿수 더하기](https://school.programmers.co.kr/learn/courses/30/lessons/120906)

### 동작 안 한 코드

```javascript
function solution(n) {
    return String(n).split('').reduce((a,b) => Number(a) += Number(b), 0)
}
```

### 동작한 코드

```javascript
function solution(n) {
  return String(n)
    .split("")
    .reduce((a, b) => (a += Number(b)), 0);
}
```

> **차이는 딱 하나.** `Number(a) += ...` vs `a += ...`.
> 왜 `Number(a)`를 붙이면 안 되냐?

### 핵심 — `Number(a)`는 "값"이지 "변수"가 아니다

```javascript
let a = 5
a += 3          // ✅ a는 변수 → 거기에 3 더해서 다시 a에 담음
Number(a) += 3  // ❌ Number(a)는 새 값 (8) → 새 값에 += 못함
```

> **`+=`의 왼쪽에는 "담을 그릇(변수)"이 와야 함.**
> `Number(a)`는 "a를 숫자로 변환한 새 값"이라서 그릇이 아니라 그냥 값임.
> `5 += 3` 이거 안 되는 거랑 똑같음. 5는 그릇이 아니니까.

### 그럼 왜 `a += Number(b)`는 됐냐?

```javascript
reduce((a, b) => (a += Number(b)), 0);
//              ↑       ↑
//           그릇O    값O  ← 정상
```

`a`는 매개변수(변수) → 그릇 OK. `Number(b)`는 그냥 값 → OK. 그릇에 값을 더하는 건 정상.

### 더 깔끔한 풀이

```javascript
function solution(n) {
  return String(n)
    .split("")
    .reduce((a, b) => a + Number(b), 0);
}
```

> `+=`는 **"a에 누적해서 다시 담는다"**는 의미인데, `reduce`는 어차피 return 값을 다음 a로 받음. 그래서 `+=`보다 그냥 `+`가 자연스러움.

### 2. `Math.sqrt` + `Number.isInteger` — 제곱수 판별

[문제: 제곱수 판별하기](https://school.programmers.co.kr/learn/courses/30/lessons/120909)

> **`Math.sqrt`는 무조건 결과를 냄** (소수든 정수든). 제곱수인지 알려면 그 결과가 정수인지를 따로 확인해야 함.

```javascript
Math.sqrt(16); // 4    ← 정수 (제곱수)
Math.sqrt(15); // 3.87... ← 소수 (제곱수 아님)
Math.sqrt(25); // 5    ← 정수 (제곱수)
```

### `Number.isInteger` — 정수인지 판별

```javascript
Number.isInteger(4); // true
Number.isInteger(3.87); // false
Number.isInteger(5); // true
```

### 두 개 합치면 끝

```javascript
function solution(n) {
  return Number.isInteger(Math.sqrt(n)) ? 1 : 2;
}
```

> **외울 패턴: 제곱수 판별 = `Number.isInteger(Math.sqrt(n))`**

### 3. `n << t` — 비트 시프트 연산 🤯

[문제: 다음에 올 숫자](https://school.programmers.co.kr/learn/courses/30/lessons/120910)

> 비트 연산자 `<<`. **왼쪽으로 시프트** = `n * 2^t`와 같음.

```javascript
1 << 3; // 8   (1 * 2³)
2 << 3; // 16  (2 * 2³)
5 << 2; // 20  (5 * 2²)
```

### 왜 빠른가?

> 컴퓨터 입장에서 비트 한 칸씩 미는 건 **곱셈보다 훨씬 빠름**. 옛날 코드에서 자주 보임.
> 근데 요즘은 가독성 때문에 `n * Math.pow(2, t)` 또는 `n * 2 ** t` 쓰는 게 일반적.

### 알아두기 정도면 OK

| 연산     | 의미          | 결과           |
| -------- | ------------- | -------------- |
| `n << t` | 왼쪽 시프트   | `n * 2^t`      |
| `n >> t` | 오른쪽 시프트 | `n / 2^t` (몫) |

> **외울 필요까진 없고 "이런 게 있구나" 정도.** 코테에서 가끔 다른 사람 풀이에 나오면 당황하지 말기.

### 4. 짝/홀 — `i++` 대신 `i+=2`

[문제: 짝수는 싫어요](https://school.programmers.co.kr/learn/courses/30/lessons/120813)

> **루프에서 if문 쓰지 말고 증감을 바꿔라.** 좋은 발상이었음.

### Before — if문으로 짝수 거르기

```javascript
function solution(n) {
  var answer = [];
  for (let i = 0; i <= n; i++) {
    if (i % 2 === 1) answer.push(i);
  }
  return answer;
}
```

### After — `i+=2`로 처음부터 홀수만

```javascript
function solution(n) {
  var answer = [];
  for (let i = 1; i <= n; i += 2) {
    answer.push(i);
  }
  return answer;
}
```

> **루프 안에서 `if`로 거르는 패턴이 보이면 → 루프 자체를 바꿀 수 있나 의심하기.**
> 시작값(`i=1`)과 증감(`i+=2`)만 조절해도 if문 통째로 사라짐.

## 어제 그 문제 — 5분 만에 해결 ✊

[문제: 두 개 뽑아서 더하기](https://school.programmers.co.kr/learn/courses/30/lessons/68644)

어제 무한루프 돌아서 1시간 헤맸던 거. 오늘 5분도 안 걸려서 찾음.

### 어제 무한루프 원인

```javascript
for(var i = 0; i < numbers.length; i++) {
    for(var j = i+1; i < numbers.length; j++) {  // ❌ i < numbers.length
        ...
    }
}
```

### 고친 거

```javascript
for(var j = i+1; j < numbers.length; j++) {  // ✅ j < numbers.length
```

> **안쪽 루프 조건이 `i`로 되어 있어서 `j`가 아무리 늘어나도 종료 조건이 안 바뀜 → 무한루프.**
> 어제 1시간 → 오늘 5분. 같은 실수 두 번째 만나니까 패턴이 보였음. **이게 성장임.**

### 이중 루프 디버깅 체크리스트

같은 실수 또 안 하려고 정리:

1. 바깥 루프 변수: `i`
2. 안쪽 루프 변수: `j`
3. **안쪽 루프 종료 조건도 `j`로 되어 있는지 확인** ← 여기 1순위
4. 증감도 `j++`인지 (`i++` 아닌지) ← 2순위

[문제: 정수 제곱근 판별](https://school.programmers.co.kr/learn/courses/30/lessons/12934)

```javascript
function solution(n) {
```

### 6. 주석으로 풀이 전략 먼저 적기 📝

[문제: 명예의 전당 (1)](https://school.programmers.co.kr/learn/courses/30/lessons/138477)

오늘 1차에 풀었음. 근데 코드 짜기 전에 **풀이 흐름을 주석으로 먼저 정리**하는 습관이 들었음. 이거 진짜 좋은 습관이라 굳히기로 함.

### 본인이 한 방식 — 의사코드(pseudocode)로 흐름 잡기

```javascript
function solution(k, scores) {
  var a = [];
  var b = [];

  for (const score of scores) {
    if (a.length < k) {
      a.push(score);
    } else {
      if (a[0] < score) {
        a[0] = score;
      }
    }

    a.sort((a, b) => a - b);
    b.push(a[0]);
  }

  return b;
}

// a변수 : k의 개수만큼의 배열의 변수를 만든다
// b변수 : return 할 배열의 변수를 만든다
// score반복문을 돌린다
// 반복문 돌리면서, a에 값을 담는다
// a에 값을 담을 때 k보다 크면, 가장 작은 애를 버린다
// 반복문 돌리면서 b에 a값 중 가장 작은 애를 담는다
// b를 리턴
```

> **이게 왜 좋은 습관이냐:** 머릿속이 꼬이기 전에 **"내가 뭘 할 건지"를 텍스트로 먼저 박아놓는 것.**
> 코드 치다가 길 잃으면 위에 적어둔 주석 보고 다시 정렬할 수 있음.
> 어제 무한루프 1시간 헤맨 거 — 이런 습관 있었으면 더 빨리 잡았을 수도 있음.

### 풀이 전략 주석 — 효율적으로 쓰는 법 (정답)

> **핵심: 코드 치기 "전에" 적고, 코드 치는 "도중에" 따라가는 것.**

### 1단계 — 코드 치기 전, 흐름만 한국어로 적기

```javascript
function solution(k, scores) {
  // 1. 명예의 전당 배열 (top): 최대 k개만 유지
  // 2. 결과 배열 (result): 매일의 최저 명예점수 기록
  // 반복:
  //   - top이 k개 미만이면 그냥 추가
  //   - 꽉 찼으면 최소값보다 큰 경우만 교체
  //   - 정렬해서 최소값을 [0]에 두기
  //   - 그날의 최저값을 result에 push
}
```

### 2단계 — 주석 따라가며 코드 채우기

```javascript
function solution(k, scores) {
  // 1. 명예의 전당 배열 (top): 최대 k개만 유지
  var top = [];
  // 2. 결과 배열 (result): 매일의 최저 명예점수 기록
  var result = [];

  // 반복:
  for (const score of scores) {
    //   - top이 k개 미만이면 그냥 추가
    if (top.length < k) {
      top.push(score);
    }
    //   - 꽉 찼으면 최소값보다 큰 경우만 교체
    else if (top[0] < score) {
      top[0] = score;
    }

    //   - 정렬해서 최소값을 [0]에 두기
    top.sort((a, b) => a - b);
    //   - 그날의 최저값을 result에 push
    result.push(top[0]);
  }

  return result;
}
```

> **주석이 코드의 뼈대가 되고, 코드가 살을 붙임.** 길 잃을 일이 없어짐.

### 풀이 전략 주석 — 4가지 꿀팁

1. **번호 매기기** — `// 1.`, `// 2.` 이렇게 번호 매기면 단계가 명확해지고 빠뜨리는 거 줄어듦.
2. **"입력 → 처리 → 출력" 순서로 적기** — 어떤 데이터가 들어와서, 어떻게 가공돼서, 뭐가 나가는지 흐름이 보임.
3. **변수 의도를 먼저 적기** — `// top: 최대 k개만 유지` 이런 식으로. 변수 만들 때 의도가 명확하면 헷갈릴 일이 줄어듦.
4. **다 푼 후엔 주석 정리하기** — 풀고 나서 너무 잘게 쪼개진 주석은 합치거나 지우기. 핵심 흐름만 남기면 나중에 봐도 이해 빠름.

### 5단계 풀이 루틴 — 굳히기

> 이 순서로 하면 헤매는 시간 진짜 줄어듦.

```
1. 문제 읽기 — 입력/출력/제약 파악
2. 풀이 전략 주석으로 적기 ← 여기가 핵심!
3. 주석 따라 코드 작성
4. 테스트 케이스 돌려보기
5. 통과 후 주석 정리 (남길 건 남기고 지울 건 지우기)
```

### 다른 사람 풀이 — `reduce` 버전

```javascript
function solution(k, score) {
  const stack = [];
  return score.reduce((a, c) => {
    if (stack.length < k) {
      stack.push(c);
      stack.sort((a, b) => a - b);
    } else {
      stack.push(c);
      stack.sort((a, b) => a - b);
      stack.shift();
    }
    a.push(stack[0]);
    return a;
  }, []);
}
```

> 일단 다 push → sort → 넘치면 shift. 본인 풀이보다 한 단계 더 함축적.
> 근데 솔직히 **본인 풀이가 더 읽기 쉬움.** `reduce`로 억지로 넣으려고 가독성 희생한 느낌.
> **"멋있어 보이는 코드 ≠ 좋은 코드".** 본인 풀이가 더 명확하면 그게 더 나음.

### 7. 문자 바꾸기 대참사 — 문자열은 불변(immutable) 🔥🔥🔥

[문제: 문자열 내 마음대로 정렬하기](https://school.programmers.co.kr/learn/courses/30/lessons/12915)

오늘의 진짜 보스. 한참 헤맴. 결국 GPT 도움 받음. 근데 **이걸 이해 못하면 자바스크립트 문자열 평생 못 다룸.**

### 시도 1 — `map`인데 안 바뀜

```javascript
strings = strings.map((string) => {
  var temp = string[0];
  string[0] = string[n];
  string[n] = temp;
});
```

### 시도 2 — `forEach`로도 안 바뀜

```javascript
strings.forEach((string) => {
  var temp = string[0];
  string[0] = string[n];
  string[n] = temp;
});
```

### 시도 3 — `for...of`도 안 바뀜

```javascript
for (let string of strings) {
  var temp = string[0];
  string[0] = string[n];
  string[n] = temp;
}
```

> **세 개 다 똑같이 안 됨.** 왜?

### 핵심 — 자바스크립트 문자열은 **불변(immutable)**

> **자바스크립트에서 문자열은 한 번 만들어지면 절대 못 바꿈.**
> `string[0] = 'a'` 이런 거 **에러도 안 나고, 그냥 무시됨.** (엄격모드에선 에러)

```javascript
let s = "hello";
s[0] = "H"; // 무시됨. 에러도 안 남.
console.log(s); // "hello"  ← 안 바뀜!
```

> **이게 자바스크립트의 함정.** 파이썬은 에러라도 내는데, 자바스크립트는 조용히 무시함. 그래서 디버깅이 더 어려움.

### 그럼 어떻게 바꿔?

> **문자열을 배열로 바꿔서 → 수정 → 다시 문자열로.**

```javascript
let s = "hello";
let arr = s.split(""); // ['h','e','l','l','o']  ← 배열은 변경 가능
arr[0] = "H"; // ['H','e','l','l','o']
let result = arr.join(""); // "Hello"
```

### `map`이 또 다른 문제

> 본인 코드에서 `map`이 안 됐던 또 다른 이유: **`map`은 return을 안 하면 새 배열에 `undefined`가 들어감.**

```javascript
strings = strings.map((string) => {
  string[0] = string[n]; // ① 문자열은 불변이라 무시됨
  string[n] = temp;
  // ② return도 없음 → undefined가 새 배열에 들어감
});
// 결과: strings = [undefined, undefined, undefined]
```

> **이중 펀치였음.** 문자열 불변 + map의 return 누락. 둘 중 하나만 있어도 망하는데 둘 다 있었음.

### 본인이 결국 만든 풀이 — 시행착오 순서대로

### 0차 — 문제를 잘못 읽음 😵

> 회고하면서 깨달았는데, **애초에 문제 해석을 잘못함.**

문제 원문: _"인덱스 1의 문자가 같은 문자열이 여럿 일 경우, 사전순으로 앞선 문자열이 앞쪽에 위치합니다."_

본인이 이해한 것:

> "n번째 글자 같으면, **swap된 상태(n번째를 앞으로 옮긴 상태)에서** 그냥 사전순으로 정렬하면 되는 거 아냐?"

실제 문제 의도:

> "n번째 글자 같으면, **원래 문자열 전체**를 사전순으로 비교."

> **이 차이 때문에 swap 두 번 하는 이상한 풀이가 나왔음.**
> 문제를 swap 기준으로 해석해버려서 → "n번째를 앞으로 옮긴 다음 그냥 sort()하면 되겠네!" 라는 발상으로 흘러간 것.
> **교훈: 문제 한 번 더 정확히 읽기. 한 단어 차이로 풀이가 완전히 달라짐.**

### 1차 시도 — 그냥 `strings.sort()`만 씀

```javascript
function solution(strings, n) {
  strings = strings.map((item) => {
    var temp = item[0];
    var arrItem = item.split(""); // ✅ 배열로 변환
    arrItem[0] = arrItem[n];
    arrItem[n] = temp;
    return arrItem.join(""); // ✅ return + 다시 문자열로
  });

  strings.sort(); // ❌ 이게 문제

  strings = strings.map((item) => {
    var temp = item[0];
    var arrItem = item.split("");
    arrItem[0] = arrItem[n];
    arrItem[n] = temp;
    return arrItem.join("");
  });

  return strings;
}
```

> **체점 결과: 통과 X.** 왜?
>
> 위에서 말한 것처럼, **문제 의도는 "원본 문자열 전체"로 사전순 비교인데**, 본인은 swap된 상태로 정렬하니까 결과가 달라진 거.
> 게다가 `sort()` 단독은 안전한 비교가 아니라서 더 꼬임.

### `strings.sort()`만 쓰면 뭐가 문제냐

> **인자 없는 `sort()`는 모든 원소를 "문자열로 변환해서" 비교함.**
> 영어 알파벳만 있으면 운 좋게 맞을 수 있는데, **n번째 글자가 같을 때 그 다음을 어떻게 처리할지 명확하지 않음.**

```javascript
["sun", "bed", "car"].sort();
// ['bed', 'car', 'sun'] — 우연히 맞아 보일 수 있음
```

근데 이 문제는 단순 사전순이 아니라 **"n번째 글자가 같으면 원본 문자열 전체를 사전순으로"**라는 규칙이 있음. 이걸 `sort()` 단독으론 표현 못함.

### 2차 시도 — GPT한테 물어봄 → `localeCompare` + 문제 의도 다시 이해

```javascript
function solution(strings, n) {
  return strings.sort((a, b) => {
    if (a[n] === b[n]) {
      return a.localeCompare(b); // n번째가 같으면 원본 전체 사전순
    }
    return a[n].localeCompare(b[n]); // n번째 글자끼리 비교
  });
}
```

> **이거면 swap도 필요 없이 한 방에 끝.**
> 문제 의도를 정확히 이해하니까 풀이도 훨씬 깔끔해짐.

### 3차 — 그래서 본인 코드에 적용해봄 (그래도 swap은 유지)

```javascript
function solution(strings, n) {
  strings = strings.map((item) => {
    var temp = item[0];
    var arrItem = item.split("");
    arrItem[0] = arrItem[n];
    arrItem[n] = temp;
    return arrItem.join("");
  });

  strings.sort((a, b) => {
    // ✅ localeCompare 적용
    if (a[0] === b[0]) {
      return a.localeCompare(b);
    }
    return a[0].localeCompare(b[0]);
  });

  strings = strings.map((item) => {
    var temp = item[0];
    var arrItem = item.split("");
    arrItem[0] = arrItem[n];
    arrItem[n] = temp;
    return arrItem.join("");
  });

  return strings;
}
```

### 본인 풀이 — 뭐가 틀렸나? (질문 답변)

> **결론: 문제 해석부터 잘못해서 swap을 끼워 넣은 게 근본 원인.**

본인 풀이는:

1. n번째 글자를 [0]으로 옮기고 (1차 swap)
2. [0] 기준 정렬
3. 다시 원위치 (2차 swap)

이렇게 한 이유 = **"n번째 같으면 swap된 상태에서 사전순"이라고 잘못 해석.**

그래서 발생한 두 가지 문제:

**문제 1 — swap한 상태로 비교함.**
원래 `"sun"`에서 `n=1`이면 `"usn"`이 됨. 이 상태로 `localeCompare`하면 `"usn"` 전체로 비교하는 거임. 본래 의도(`"sun"` 전체로 비교)와 결과가 달라질 수 있음.

**문제 2 — 굳이 원본 건드릴 필요 없음.**
문제 의도를 정확히 읽었으면 swap 없이 **비교할 때만 n번째 인덱스를 들여다보면 됨**:

```javascript
function solution(strings, n) {
  return strings.sort((a, b) => {
    if (a[n] === b[n]) {
      return a.localeCompare(b);
    }
    return a[n].localeCompare(b[n]);
  });
}
```

> **`sort`의 비교 함수는 "원본을 안 건드리고 어떤 순서로 놓을지만 결정".**
> 그래서 swap 같은 거 할 필요 없이, **"비교할 때만 n번째 기준으로 본다"**가 정답.
> 이게 정렬을 한 단계 더 이해한 사고방식.

### 오늘의 진짜 깨달음 — 문제 정확히 읽기 + `sort()` 단독은 함정

> **1순위:** 문제를 한 단어씩 정확히 읽기. "원본 문자열" vs "swap된 문자열" — 한 끗 차이로 풀이가 완전히 달라짐.
> **2순위:** 영어만 다루면 `sort()` 단독으로도 어느 정도 맞아 보여서 더 헷갈림. 하지만 **"이런 규칙으로 정렬하라"** 같은 조건이 있으면 **무조건 비교 함수를 직접 써야 함.**
> 비교 함수 안에서 어떻게 비교할지 직접 코드로 표현하는 게 진짜 정렬임.

---

### 8. `localeCompare` — 문자열 비교 정석 🆕

> **두 문자열을 사전순으로 비교해서 -1, 0, 1을 반환.**

```javascript
"apple".localeCompare("banana"); // -1 (apple이 먼저)
"banana".localeCompare("apple"); //  1 (banana가 뒤)
"apple".localeCompare("apple"); //  0 (같음)
```

### `sort`의 규칙 복습

| 반환값    | 의미             |
| --------- | ---------------- |
| 음수 (-1) | a가 b보다 앞     |
| 0         | 같음 (순서 유지) |
| 양수 (1)  | a가 b보다 뒤     |

> **`localeCompare`가 정확히 이 형식으로 반환.** 그래서 `sort` 안에 그대로 박아 넣으면 됨.

### 그냥 `<`, `>` 쓰면 안 되나?

쓸 수는 있는데 **한글, 악센트 문자, 대소문자** 등 처리가 부정확함.

```javascript
// ❌ 단순 비교 — ASCII 코드 기준
"가".localeCompare("나"); // -1 (정확히 한국어 사전순)
"가" < "나"; // true (운 좋게 맞음)

"é".localeCompare("f"); // -1 (정확)
"é" < "f"; // false (틀림! 유니코드 코드포인트가 더 큼)
```

> **영어/한글만 다룰 거면 단순 비교도 됨.** 근데 `localeCompare`가 더 정확하고 의도가 명확함.
> **외울 패턴:** 문자열 정렬은 `localeCompare`.

---

### 9. `(a > b) - (a < b)` — 천재적인 정렬 트릭 🆕

> **이거 처음 보면 "뭐지?" 싶은데, 알면 평생 써먹음.**

### 동작 원리

자바스크립트에서 `true`는 1, `false`는 0으로 자동 변환됨.

```javascript
true - false; // 1 - 0 = 1
false - true; // 0 - 1 = -1
true - true; // 1 - 1 = 0
```

### 그래서 `(a > b) - (a < b)`는?

| 상황     | `a > b`   | `a < b`   | 결과   |
| -------- | --------- | --------- | ------ |
| a가 큼   | true (1)  | false (0) | **1**  |
| a가 작음 | false (0) | true (1)  | **-1** |
| 같음     | false (0) | false (0) | **0**  |

> 정확히 `sort`가 원하는 -1 / 0 / 1 형식.

### 사용 예

```javascript
// 숫자 오름차순
arr.sort((a, b) => (a > b) - (a < b));

// 위와 동일하지만 더 짧게
arr.sort((a, b) => a - b); // 숫자라면 이게 더 간단

// 문자열 오름차순
arr.sort((a, b) => (a > b) - (a < b)); // ← 여기서 진가
```

### 언제 쓰면 좋냐?

> **숫자가 아닌 비교, 또는 `localeCompare` 안 써도 되는 간단한 문자열 비교에 유용.**
> 코테에서 다른 사람 풀이에 자주 보임. 모르면 당황하니까 외워두기.

```javascript
// "정수가 아닌 값 정렬"의 만능 패턴
sort((a, b) => (a > b) - (a < b));
```

### 주의점

> 한글/유니코드 정확한 정렬이 필요하면 `localeCompare`가 안전함.
> 단순 비교 정도면 `(a > b) - (a < b)`도 OK.

```javascript
s = "Hello"; // ✅ 새 문자열 할당은 OK

// 글자를 바꾸려면 → 배열 거쳐서

```

### `map` 사용 시 체크리스트

```javascript
arr.map((item) => {
  // 1. return 있는지?           ← 없으면 [undefined, ...] 됨

```

### `+=`의 왼쪽엔 변수만

```javascript
a += 3            // ✅ a는 변수

```

### 정렬 — 무기 3종 세트

```javascript
arr.sort((a, b) => a - b); // 숫자 오름차순
arr.sort((a, b) => a.localeCompare(b)); // 문자열 (정확)
arr.sort((a, b) => (a > b) - (a < b)); // 만능 트릭
```

```javascript
Number.isInteger(Math.sqrt(n)); // true면 제곱수
```

```javascript
  }
```
